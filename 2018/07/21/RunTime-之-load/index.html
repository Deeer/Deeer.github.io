RunTime 之 +load

发表于 2018-07-21   |   分类于 ScourceCode

起始

在系统进入的main函数之前，会先处理的项目文件中的+load方法。
做法主要分为两个部分：
1.加载+load方法
2.调用+load方法

代码中我们能看到，系统在查找时也会去判断是否已经加载过，如果没有则在加载钱调用递归互斥锁，防止引起I/O问题。

预加载

接下来的我们来看下加载方法

这里主要看以下几个方法

_getObjc2NonlazyClassList

这个方法实际上使用宏重载的方式批量创建的方法，主要作用是通过文件对象字段返回classref类型的结构体指针

remapClass

这个方法返回的是一个Class,它返回的可能是已经被重新分配过的类结构体，它会从hash表中去匹配查找是否存在,找到后返回这个对象。
这里使用了NXMapTable来记录相关的数据，其中NXMapTable包含以下属性

  • _NXMapTablePrototype: 包含三个结构体指针,用于表中的数据处理和比较
  • count 记录表中数据数量
  • buckets 具体的表数据数组。其中包含的_MapPair结构如下

具体结构如图

  • nbBucketsMinusOne: 未做深入了解

schedule_class_load

这个方法中调用了schedule_class_load(cls->superclass);，这使得父类的+load方法总是先于子类被添加，也就使得父类的+load方法总是先于子类被调用

同时在这方法中，还会处理添加类时内存不足的问题，如果分配的内存空间已经被使用了超过一半，则调用realloc方法进行重新分配

_getObjc2NonlazyCategoryList

同样的，这个方法是去获取所有category数据,并进行加载。这里的调用顺序，也恰好证实了，类中+load方法调用总是会先于分类的。

调用

call_load_methods

为了优化内存，这里手动创建了一个自动释放池。降低了do-while时产生的内存消耗。我们也能很清楚的看到，实际上category和class中的+load方法是分开调用的，因为他们分别存放在了两个不同的数据结构中，并且，runtime会尽可能的先调用class中的+load方法，直到其中的方法调用完之后，再调用category中的方法。

总结：

  • 本文介绍了+load方法的加载方式和调用顺序，并从代码角度解释了父类中的+load方法总是先于子类被调用。本类中的份+load方法总是先于分类被调用
  • 介绍了加载时，用到的Class数据存储结构NXMapTable,已经具体的存储结构
